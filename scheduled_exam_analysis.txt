               ভাগ ১: বর্তমান Data Architecture (বোঝা দরকার)
================================================================================

Google Sheet-এ "batches" শিটে একটি কলাম আছে: `assignedItemsMap`

এটি একটি JSON string হিসেবে store হয়:
    { "itemId_abc": "2025-05-01T10:00:00Z", "itemId_xyz": "2025-05-15T08:00:00Z" }
    key   = Library item-এর id
    value = কখন শেয়ার করা হয়েছে (assigned timestamp, ISO string)

GAS `apiShareLibraryItem(itemId, batchIdsMap)` ফাংশনটি:
    - batchIdsMap[batchId] === true  → assigned[itemId] = new Date().toISOString()
    - batchIdsMap[batchId] === false → delete assigned[itemId]

Student Library পড়ার সময়:
    const assignedItemsMap = studentBatch.assignedItemsMap || {};
    const assignedIds = Object.keys(assignedItemsMap); // শুধু keys নেয়, value ব্যবহার হয় না

মূল পয়েন্ট: value হিসেবে যে ISO string আছে, সেটা StudentLibrary.tsx-এ
আদৌ ব্যবহার হয় না — শুধু key (itemId) গুলো দরকার।

────────────────────────────────────────────────────────────────────────────────
সমস্যা #1 ⚠️ [CRITICAL] — Data Format Breaking Change
────────────────────────────────────────────────────────────────────────────────
Gemini বলেছে: "assignedItemsMap-এর ভেতর scheduled time store করা হবে।"

এর মানে হলো value-টি পরিবর্তন করা:
    BEFORE: "itemId_abc" → "2025-05-01T10:00:00Z"   (simple string)
    AFTER:  "itemId_abc" → { assignedAt: "...", scheduledStartTime: "..." }  (object)

    অথবা: "itemId_abc" → "2025-06-14T08:00:00Z||2025-05-01T10:00:00Z"  (concatenated)

এই পরিবর্তনে কী ভাঙবে?

Google Sheet-এ এখন যে batches আছে, তাদের `assignedItemsMap` cell-এ পুরনো
format-এর data আছে (simple string value)। নতুন code যদি object/JSON
format expect করে কিন্তু পুরনো string পায়, তখন:

    var scheduled = assigned[itemId].scheduledStartTime;
    // ❌ TypeError: Cannot read property 'scheduledStartTime' of string

ফলে সব batch-এর সব পুরনো assignment crash করবে।
StudentLibrary সম্পূর্ণ blank হয়ে যাবে।
সব ছাত্র-ছাত্রী কিছুই দেখতে পাবে না।

Gemini পুরনো data handle করার কোনো mention করেনি। এটাই সবচেয়ে বড় ঝুঁকি।

সমস্যা #4 ℹ️ [Low] — Countdown Timer এর setInterval cleanup

      if (item.type !== 'exam') return { locked: false, scheduledTime: null };
      const timeStr = scheduledStartTimeMap[item.id];
      if (!timeStr) return { locked: false, scheduledTime: null };
      const scheduledTime = new Date(timeStr);
      return {
        locked: new Date() < scheduledTime,
        scheduledTime
      };
    }

React state:

    const [lockedExamTime, setLockedExamTime] = useState<Date | null>(null);
    const [countdown, setCountdown] = useState('');

Countdown effect:
    useEffect(() => {
      if (!lockedExamTime) return;

      const tick = () => {
        const now = new Date();
        const diff = lockedExamTime.getTime() - now.getTime();

        if (diff <= 0) {
          // Exam unlocked! Modal বন্ধ করুন, exam খুলুন
          setLockedExamItem(null);
          setLockedExamTime(null);
          setCountdown('');
          // তারপর স্বাভাবিক exam open করুন
          return;
        }

        const h = Math.floor(diff / 3600000);
        const m = Math.floor((diff % 3600000) / 60000);
        const s = Math.floor((diff % 60000) / 1000);
        setCountdown(`${h}:${String(m).padStart(2,'0')}:${String(s).padStart(2,'0')}`);
      };

      tick(); // সাথে সাথে একবার চালান
      const interval = setInterval(tick, 1000);
      return () => clearInterval(interval); // ← অবশ্যই cleanup
    }, [lockedExamTime]);

               ভাগ ৫: এটা করা উচিত কিনা? Pros ও Cons
================================================================================

PROS ✅ (কেন করবেন):
─────────────────────
1. একটি exam upload করে সব batch-কে আলাদা time-এ দেওয়া যাবে।
   Admin-এর অনেক সময় বাঁচবে।

2. Student প্রশ্ন দেখতে পাবে না time-এর আগে। Fair exam নিশ্চিত হবে।

3. Countdown timer দেখে student জানতে পারবে কখন exam শুরু হবে।
   Confusion কমবে।

4. Data model change minimal (নতুন একটা field মাত্র)।
   Quota impact: শূন্য — বিদ্যমান getBatches() read-এ এই field-ও আসে।

5. Feature remove করা সহজ — scheduledStartTimeMap empty রাখলেই হয়।

CONS ❌ (ঝুঁকি ও সীমাবদ্ধতা):
───────────────────────────────
1. Client-side enforcement only:
   একজন tech-savvy ছাত্র browser dev tools খুলে JavaScript modify করে
   বা device clock change করে bypass করতে পারে। সার্ভার-সাইড lock নেই।
   ➤ কিন্তু এটা private tuition (২০-৫০ জন trusted student), তাই acceptable।

2. Device time mismatch:
   যদি student-এর phone-এর time ভুল থাকে, exam আগেই বা পরেই unlock হবে।
   ➤ Solution: Countdown শুধু দেখানোর জন্য। Actual check server timestamp-এ
     করা ideal, কিন্তু এই app-এ GAS server call avoid করা উচিত (quota)।
     Client-side time-ই use করতে হবে।

3. Implementation complexity:
   Admin Library share modal-এ datetime picker যোগ করলে UI জটিল হবে।
   ভুলভাবে করলে datetime local timezone conversion-এ bug আসতে পারে।
   ➤ datetime-local input সবসময় local time দেয়। UTC-তে convert করে
     save করতে হবে: `new Date(input.value).toISOString()`

4. Existing shared exams-এ scheduled time থাকবে না:
   আগে থেকে batch-এ assign করা exams-এ scheduledStartTimeMap-এ কিছু
   নেই, তাই সেগুলো immediately accessible থাকবে। এটা correct behavior।
   কিন্তু admin-কে জানাতে হবে যে পুরনো exams-এ time add করতে হলে
   আবার share করতে হবে।
